Configuration Management and the Cloud week 4

storing data in the cloud
block storage: acts like a hard drive. can be persistent or ephemeral. another system may access the data through NFS or CIFS.
object/blob storage: lets you place and retrieve objects in a storage bucket, and requires no file system. may need an API to access the data.
blob: binary large object.

SQL database: relational. use the traditional database format and query language. retrieve the data by writing SQL queries.
No SQL database: ease of development, functionality, and performance at scale, but need to use a specific API (might need to rewrite the app).
https://aws.amazon.com/nosql/?nc1=h_ls

storage performant factors
throughput: like gigabytes per second.
IOPS: input/output operation per second.
latency: time to the first byte or time it takes a write operation to complete

hot data: accessed frequently.
cold data: accessed infrequently.

load balancing
round robin DNS: client is served in turn. giving the client a list of IPs in a different order. the client will try addresses in succession.
dedicated load balancer: directs clients to the back-end server based on the rule we provided
sticky sessions: the same client always goes to the same backend server.
performing health checks: stop sending new requests to the unhealthy server.

content delivery networks(CDNs): caching content physically close to the user.

change management:
1. well testing: unit tests and integration tests. build a continuous integration(CI) system to test code every time there is a change.
CI system: Jenkins, Travis CI, or CI as a service provided by a cloud vendor.
2. continuous deployment(CD) system: push changes to other environments (like a test, development, and pre-prop environment) before pushing them to the production(prop) environment.
3. A/B testing: serve requests in two different sets of code and configuration, monitoring the performance or debugging. can separate two groups by a load balancer.

quotas or limits
rate limits: limits calls to prevent overloading.
utilization limits: limits the total amount of a certain resource.
https://cloud.google.com/compute/quotas#understanding_vm_cpu_and_ip_address_quotas

monitoring and alerting
metric: a descriptive statistic, indicator, or figure of merit.
can be specific like HTTP response codes or purchases success rate.
monitoring tool provided by cloud vendors or other systems like Prometheus, Datadog, and Nagios.
these systems might use pull or push models.
https://www.digitalocean.com/community/tutorials/an-introduction-to-metrics-monitoring-and-alerting
https://www.datadoghq.com/blog/monitoring-101-collecting-data/

whitebox monitoring: check the system from the inside.
blackbox monitoring: check the system from the outside.

divide alerts into urgent (by pager, etc.. also calls paging alert) and non-urgent (by tickets, etc.)
all alerts should be actionable.

service-level objectives (SLOs): pre-established performance goals for service. these goals should be measurable. e.g., SLO of 99% availability means it can be down 1% of the time. 99% availability service also call a two 9 service. 99.999% call a five 9 service. three 9 service is fine for a lot of IT systems. SLOs are more like soft targets.

services providers usually have promises in the form of service level agreements (SLAs). a commitment between a provider and client.

use SLOs and SLAs to help you decide on monitors and alerts setting.

error budget: a three 9 service can have 43 minutes error budget per month.
https://en.wikipedia.org/wiki/High_availability

troubleshooting clouds
1. knowing where can you find logs and what info is available.
2. doing rollbacks.
3. create a snapshot of the disk image and mount it on a healthy machine.
4. deploy the same container as the cloud to another machine.
5. isolate the faulty behavior.
6. bring your service to another region or diffident machine type.
7. call the service provider.
8. back up data, automatic backups, and check periodically.
9. a wall-documented recovery plan.
10. prepare a secondary instance.
https://cloud.google.com/compute/docs/troubleshooting/troubleshooting-vm-creation
https://sre.google/books/
